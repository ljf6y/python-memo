① 使用 DB 和 schema

原文大意：

使用 DB 和 schema
OSC_5_04_01_0.sql 这个 SQL 里，数据库名和 schema 名写死了，
这是不是应该做成可变的比较好？
另外，在 VSCode 上和 Snowsight 上显示的路径不一样，
是不是应该把 Snowsight 上的状态 当作证据？
按照上面说的，这个 SQL 里已经写死了，所以感觉它并没有作为“设置文件”发挥作用。

中文解释
领导的意思是：
	•	你这个 SQL 里 DB 名、schema 名是固定写死的
	•	他在怀疑：
“既然是设置文件，为什么不是可变参数，而是直接写死？”
	•	同时他还在说：
“VSCode 上看到的路径和 Snowsight 上看到的不一样，那到底应该以哪个为准？”
	•	他的倾向是：
最好拿 Snowsight 里的状态做证据
	•	最后一句是在质疑：
“如果 SQL 里都已经写死了，那它看起来就不像‘配置文件’，而只是一个固定 SQL。”

简单说
他在问你两个点：
	1.	DB/schema 为什么写死？
	2.	证据应该以 VSCode 还是 Snowsight 为准？

② 证据 No 栏

最初のエビデンスのNo欄に「1」を記載してください。

意思就是：

这个比较简单，就是格式修正。

③ No.7 的阈值写错了

原文：

No.7
網羅性パターン表のNo.20とNo.24は年月の境界値チェックとなるはずなのでそれぞれ190001と210101となるのが適切です。

中文解释
意思是：

“网罗性模式表里的 No.20 和 No.24，应该是‘年月的边界值检查’，所以分别写成 190001 和 210101 才合适。”

这里他在说什么
你这个检查对象应该是 年月（YYYYMM） 的边界值。

所以边界值测试通常会写成这种感觉：
	•	下限侧：190001
	•	上限侧：210101（这个你要再结合设计书确认一下，他这里是这么写的）

也就是说领导觉得你 No.20、No.24 这两个测试值设置得不对，应该改成更符合“边界值检查”的数据。

⸻

④ 看不出哪个是处理前、哪个是处理后

原文：

エビデンスを見てもどれが処理前でどれが処理後でどうなるのかという説明を入れてください。
このシートに関しては理解できてないのでレビュー未実施と同じです。

中文解释
这个比较重。

第一句意思：

“光看证据，没法知道哪个是处理前、哪个是处理后，以及最终会变成什么样，请补充说明。”

第二句意思：

“关于这张 sheet，我现在无法理解，所以跟没有做 review 基本一样。”

简单说
领导是在说：
	•	你这张证据 说明不够
	•	他看不懂：
	•	哪张图/哪部分是 处理前
	•	哪张图/哪部分是 处理后
	•	经过处理后 结果应该怎么变化
	•	因为看不懂，所以他认为：
这张表等于没法 review

这个指摘算比较严厉，不只是小修正，而是说：
证据的可读性不够，别人无法靠它完成审查。

⸻

4 个指摘总结成最简单的话
	1.	SQL 里的 DB/schema 为什么写死？设置方式是否合理？证据应该以 Snowsight 为准吧？
	2.	第一张证据的 No 栏要写 1
	3.	No.20、No.24 的边界值写错了，要改成更合适的年月边界值
	4.	证据没写清楚处理前/处理后，导致他看不懂，等于没法 review
